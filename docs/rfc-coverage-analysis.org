#+TITLE: RFC Coverage Analysis for http-axiom (hax)
#+AUTHOR: Vikram Shah (Security Research)
#+DATE: 2026-03-26
#+OPTIONS: toc:3 num:t

* Executive Summary

hax claims to be "RFC-grounded" --- every predicate should trace to a specific
RFC section. This document audits that claim. Of 21 predicates across 5 groups,
only 2 cite specific RFC sections in their implementation (HSTS and
idempotency/safety). The remaining 19 are grounded in web security best
practices rather than normative RFC text. Several predicates are stubs (return
"skip" unconditionally). The cache and state groups are particularly
underdeveloped relative to the RFC surface they should cover.

Beyond citation gaps, significant RFC territory remains unaddressed: HTTP
semantics (RFC 9110) has approximately 40 testable behavioral requirements of
which hax covers 3; HTTP caching (RFC 9111) has approximately 25 of which hax
covers fragments of 4; and the cookie specification (RFC 6265bis) defines
roughly 15 security-relevant attributes of which hax checks 3.

* Methodology

1. Read every predicate implementation in =internal/predicate/=.
2. Extract explicit RFC citations from code comments.
3. Map each predicate to the normative specification it should be testing.
4. Compare actual checks against what the cited specification requires.
5. Identify specification territory that has no corresponding predicate.
6. Survey prior art in HTTP compliance testing for structural comparison.

** Implementation Depth Classification

- *Functional*: Parses the header/response and checks semantic properties
  (10 predicates)
- *Presence*: Checks whether a header exists but does not validate its value
  meaningfully (4 predicates)
- *Stub*: Returns "skip" unconditionally, no implementation
  (6 predicates)
- *Partial*: Some checking but significant gaps relative to the specification
  (5 predicates)

* Per-Predicate RFC Analysis

** Headers Group

*** csp (Content-Security-Policy)

*Normative specification*: W3C Content Security Policy Level 3
(https://www.w3.org/TR/CSP3/)

*What hax checks*:
- Header presence
- =unsafe-inline= + =unsafe-eval= co-occurrence in =script-src= / =default-src=
- =unsafe-inline= alone
- Wildcard sources (=*=, =data:=, =blob:=, =https:=)
- Missing =default-src=

*What the spec requires that hax does NOT check*:
- =report-uri= / =report-to= directive validity
- =base-uri= restriction (XSS via =<base>= tag injection)
- =form-action= restriction (data exfiltration via form submission)
- =frame-ancestors= vs =X-Frame-Options= consistency
- =object-src= restriction (Flash/Java plugin exploitation)
- =require-trusted-types-for= directive
- =strict-dynamic= interaction with nonce/hash (partially negates =unsafe-inline=)
- Multiple CSP header semantics (intersection of policies)
- Meta-element =<meta http-equiv="Content-Security-Policy">= vs header
- Nonce randomness and length requirements
- =sandbox= directive evaluation

*Assessment*: Good for a first pass. The script-src focus is the highest-priority
check. Missing =base-uri= and =form-action= are the most security-critical gaps.

*** hsts (Strict-Transport-Security)

*Normative specification*: RFC 6797 (HTTP Strict Transport Security)

*What hax checks*:
- Header presence
- =max-age= numeric value parsing (RFC 6797 S6.1.1 cited)
- =max-age=0= detection (cited as disabling HSTS)
- =max-age < 31536000= warning (1 year minimum)
- =includeSubDomains= presence (RFC 6797 S6.1.2 cited)

*What the spec requires that hax does NOT check*:
- RFC 6797 S7.2: HSTS host must not be accessed over plain HTTP first
  (bootstrapping problem --- HSTS preload list addresses this)
- RFC 6797 S8.1: Only effective over HTTPS (hax should warn if target is HTTP)
- RFC 6797 S8.4: HSTS and IP addresses (HSTS must not apply to IP-literal hosts)
- =preload= directive (not in RFC but universally relevant; see hstspreload.org)
- Multiple HSTS headers (should be only one per RFC 6797 S6.1)

*Assessment*: Best-cited predicate in the project. The RFC section references in
code comments are correct and precise. The missing checks are secondary.

*** samesite (Cookie SameSite attribute)

*Normative specification*: RFC 6265bis (Cookies: HTTP State Management Mechanism)
S5.4.7, S8.8

*What hax checks*:
- Presence of =Set-Cookie= headers
- =SameSite= attribute parsing per cookie
- =SameSite=None= without =Secure= flag (browser rejection)
- =Secure= and =HttpOnly= flag co-presence
- Unrecognized SameSite values

*What the spec requires that hax does NOT check*:
- RFC 6265bis S4.1.2.5: =__Secure-= prefix requirements (must have Secure flag)
- RFC 6265bis S4.1.2.5: =__Host-= prefix requirements (Secure, no Domain, Path=/)
- RFC 6265bis S5.4: =Domain= attribute scope (domain mismatch allows subdomain
  cookie injection)
- RFC 6265bis S8.5: =Secure= flag required for all session cookies (not just
  SameSite=None)
- =Max-Age= / =Expires= analysis (persistent vs session cookies)
- RFC 6265bis S8.6: Cookie size limits (4096 bytes per cookie)
- Multiple =Set-Cookie= with same name (last-writer wins ambiguity)

*Assessment*: Good per-cookie attribute parsing. Missing the cookie-prefix
security mechanism (=__Secure-= / =__Host-=) is a notable gap since these are
the strongest cookie security feature available.

*** corp (Cross-Origin-Resource-Policy)

*Normative specification*: Fetch Living Standard, "Cross-Origin-Resource-Policy"
header (https://fetch.spec.whatwg.org/#cross-origin-resource-policy-header)

*What hax checks*:
- Header presence only

*What the spec requires that hax does NOT check*:
- Valid values: =same-site=, =same-origin=, =cross-origin=
- Interaction with CORS (CORP is evaluated before CORS)
- CORP is only meaningful for subresource requests, not navigation

*Assessment*: Presence-only check. Should validate the value is one of the three
defined values, analogous to how COEP and COOP are checked.

*** x-frame-options

*Normative specification*: RFC 7034 (HTTP Header Field X-Frame-Options),
superseded by CSP =frame-ancestors= per RFC 7762

*What hax checks*:
- Header presence
- Falls back to CSP =frame-ancestors= if header absent
- Valid values: =DENY=, =SAMEORIGIN=
- Warns on =ALLOWALL=

*What the spec requires that hax does NOT check*:
- RFC 7034 S2.1: =ALLOW-FROM= with URI (deprecated but still seen)
- Conflict between X-Frame-Options and CSP =frame-ancestors= (CSP takes
  precedence per CSP3 spec)
- Multiple X-Frame-Options headers (ambiguous behavior across browsers)
- Case sensitivity of values (RFC 7034 says case-insensitive; hax uppercases
  correctly)

*Assessment*: Adequate. The fallback to CSP frame-ancestors is a good design
choice. Should warn when both X-Frame-Options and frame-ancestors are present
with conflicting semantics.

*** x-content-type-options

*Normative specification*: Originally a Microsoft extension; codified in Fetch
Living Standard S3 "X-Content-Type-Options header"

*What hax checks*:
- Header presence
- Value is =nosniff= (case-insensitive)

*What the spec requires that hax does NOT check*:
- Only meaningful for =script= and =style= MIME types (when nosniff is set, the
  browser refuses to load a resource if the MIME type does not match)
- Interaction with =Content-Type= header (nosniff without Content-Type is less
  useful)

*Assessment*: Adequate for the header's narrow scope. Could add a contextual
warning when =Content-Type= is absent alongside =nosniff=.

*** permissions-policy

*Normative specification*: W3C Permissions Policy
(https://www.w3.org/TR/permissions-policy-1/)

*What hax checks*:
- Header presence
- Fallback to legacy =Feature-Policy= header

*What the spec requires that hax does NOT check*:
- Directive syntax validation (=camera=(), microphone=()=, etc.)
- Structured header format (RFC 8941 Structured Field Values)
- Known directive names vs unknown/misspelled directives
- Wildcard =*= usage in allowlists
- Iframe =allow= attribute interaction
- Default allowlist behavior per feature

*Assessment*: Presence-only. The most under-implemented of the header group.

*** referrer-policy

*Normative specification*: W3C Referrer Policy
(https://www.w3.org/TR/referrer-policy/)

*What hax checks*:
- Header presence
- Classifies known values into safe/unsafe/warning categories
- Warns on =unsafe-url= and =no-referrer-when-downgrade=
- Reports unrecognized values

*What the spec requires that hax does NOT check*:
- Multiple comma-separated values (fallback list; last supported value wins)
- Interaction with =<meta name="referrer">= element
- Per-element =referrerpolicy= attribute override
- Empty string value (defers to a higher-level policy)

*Assessment*: Good value classification. Missing the comma-separated fallback
list parsing is the most significant gap.

*** coep (Cross-Origin-Embedder-Policy)

*Normative specification*: HTML Living Standard "Cross-Origin-Embedder-Policy"
(https://html.spec.whatwg.org/multipage/origin.html#coep)

*What hax checks*:
- Header presence
- Valid values: =require-corp=, =credentialless=
- Warns on unrecognized values

*What the spec requires that hax does NOT check*:
- Interaction with CORP and CORS (COEP requires embedded resources to opt in)
- Cross-origin isolation tuple (COEP + COOP together enable
  =crossOriginIsolated=)
- =require-corp= vs =credentialless= security trade-offs

*Assessment*: Adequate for a standalone check. Should be tested in combination
with COOP for the cross-origin isolation property.

*** coop (Cross-Origin-Opener-Policy)

*Normative specification*: HTML Living Standard "Cross-Origin-Opener-Policy"
(https://html.spec.whatwg.org/multipage/origin.html#cross-origin-opener-policies)

*What hax checks*:
- Header presence
- Valid values: =same-origin=, =same-origin-allow-popups=
- Warns on =unsafe-none= (default, no isolation)
- Warns on unrecognized values

*What the spec requires that hax does NOT check*:
- =same-origin-plus-COEP= value (newer, enables cross-origin isolation)
- Browsing context group switching behavior
- Report-only mode (=Cross-Origin-Opener-Policy-Report-Only=)
- Interaction with COEP for full cross-origin isolation

*Assessment*: Adequate. Same note as COEP --- the combined COEP+COOP check is
the real security property.

** Methods Group

*** idempotency

*Normative specification*: RFC 9110 S9.2.2 (Idempotent Methods)

*What hax checks*:
- Sends PUT twice with identical body
- Reads state via GET after each PUT
- Compares response bodies (state, not status codes)
- Correctly cites RFC 9110 S9.2.2

*What the spec requires that hax does NOT check*:
- RFC 9110 S9.2.2 defines PUT and DELETE as idempotent; hax only tests PUT
- HEAD, GET, OPTIONS, TRACE are also idempotent (trivially, since they are safe)
- The spec says "excluding error or expiration issues" --- hax does not account
  for this caveat (transient errors could cause false positives)
- DELETE idempotency is more nuanced: DELETE of a nonexistent resource may
  return 404, which is correct idempotent behavior (the resource state is
  "deleted" in both cases)

*Assessment*: The state-comparison approach (instead of status-code comparison)
is architecturally sound and matches the RFC intent. Should extend to test
DELETE idempotency. The haxgoat server intentionally violates DELETE idempotency
but hax cannot currently test it.

*** safety

*Normative specification*: RFC 9110 S9.2.1 (Safe Methods)

*What hax checks*:
- Reads state via GET before and after a test GET
- Compares response bodies
- Correctly cites RFC 9110 S9.2.1

*What the spec requires that hax does NOT check*:
- HEAD, OPTIONS, TRACE are also safe methods
- RFC 9110 S9.2.1: "safe methods are not expected to cause any change" ---
  the word "expected" introduces an intent qualifier; logging and counter
  increments may be considered acceptable side effects
- The response body comparison is a heuristic; a truly safe method could return
  different representations (time-varying data, random content) while causing
  no state change

*Assessment*: The body-comparison heuristic works for haxgoat but may have false
positives in production where responses contain timestamps or request counters.
Could be improved by repeating the GET multiple times and checking monotonicity
of the response variations.

*** retries

*Normative specification*: RFC 9110 S9.2.2 (implicit), RFC 9110 S15.4
(automatic retry on connection failure)

*What hax checks*:
- Nothing. Returns "skip" unconditionally.

*What the spec requires*:
- RFC 9110 S9.2.2: Idempotent methods can be automatically retried
- RFC 9110 S9.2.1: Safe methods can be automatically retried
- Non-idempotent methods (POST) should not be automatically retried
- RFC 9110 S15.4: Handling of retries after 5xx status codes
- RFC 8470 (Using Early Data in HTTP): Replay risk with 0-RTT TLS data;
  servers should respond 425 Too Early to prevent unsafe replay

*Assessment*: Completely unimplemented. The retry predicate should verify that
POST requests are not silently retried and that servers properly handle
repeated idempotent requests.

** Cross-Origin Group

*** csrf

*Normative specification*: No single RFC; CSRF protection is defined across:
- RFC 6265bis S8.8 (SameSite cookie attribute)
- OWASP CSRF Prevention Cheat Sheet
- Fetch Living Standard (CORS preflight for non-simple requests)

*What hax checks*:
- Nothing. Returns "skip" unconditionally.

*What the spec requires*:
- Verify that state-changing endpoints reject requests without valid CSRF tokens
- Verify that SameSite=Strict/Lax cookies prevent cross-origin state mutation
- Verify that custom headers (e.g., =X-CSRF-Token=) are required for mutations
- Verify that preflight (OPTIONS) is required for non-simple cross-origin
  requests

*Assessment*: Completely unimplemented. CSRF is mapped to OWASP A01 (Broken
Access Control) and is one of the most critical security properties at L1.
The test structure requires: (1) GET a form/token, (2) POST without the token,
(3) verify rejection.

*** cors

*Normative specification*: Fetch Living Standard S3.2 (CORS protocol)
(https://fetch.spec.whatwg.org/#http-cors-protocol)

*What hax checks*:
- =Access-Control-Allow-Origin= value (wildcard, null, specific origin)
- =Access-Control-Allow-Credentials= in combination with wildcard
- Warns on null origin (sandboxed iframe bypass)

*What the spec requires that hax does NOT check*:
- Preflight request/response validation (OPTIONS method handling)
- =Access-Control-Allow-Methods= validation
- =Access-Control-Allow-Headers= validation
- =Access-Control-Expose-Headers= (which response headers are readable)
- =Access-Control-Max-Age= (preflight cache duration)
- Preflight caching correctness
- Credentialed vs non-credentialed request semantics
- Simple request criteria (when preflight is not required)
- The distinction between same-origin, same-site, and cross-origin
- Request header restrictions (forbidden headers)

*Assessment*: Covers the highest-severity CORS misconfigurations (wildcard,
null, credential combo). Missing preflight validation is the largest gap ---
many CORS bypasses exploit improper preflight handling.

*** cors-reflection

*Normative specification*: Fetch Living Standard (same as cors above)

*What hax checks*:
- Whether server echoes the sent =Origin= header into =Access-Control-Allow-Origin=
- Detects credentialed reflection (=ACAO= + =ACAC: true=)

*What the spec requires that hax does NOT check*:
- Partial reflection (e.g., =*.example.com= pattern matching)
- Regex-based origin validation bypasses (e.g., =evil-example.com=)
- Null origin reflection (covered by cors predicate separately)
- Subdomain-based reflection (=sub.target.com= reflected when =target.com= is
  allowed)

*Assessment*: Good for the exact-reflection case. Missing the regex/pattern
bypass variants is significant --- in practice, most CORS reflection bugs are
partial-match issues, not exact reflection.

*** jsonp

*Normative specification*: No RFC; JSONP is a legacy pattern. Security
implications documented in various advisories.

*What hax checks*:
- Nothing. Returns "skip" unconditionally.

*What the spec requires*:
- Detect endpoints that accept a =callback= parameter and wrap JSON in a
  function call
- Verify that JSONP endpoints require authentication/authorization
- Detect JSONP on state-reading endpoints (information disclosure)

*Assessment*: Low priority. JSONP is a legacy technology superseded by CORS.
However, many APIs still support it for backward compatibility, making it a
valid check.

*** redirect

*Normative specification*: RFC 9110 S15.4 (Redirection 3xx)

*What hax checks*:
- Detects 3xx status codes
- Warns if =Location= header points to =http://= (insecure redirect)

*What the spec requires that hax does NOT check*:
- RFC 9110 S15.4.2 (301): Permanent redirect --- method may change
- RFC 9110 S15.4.3 (302): Found --- method may change (historical)
- RFC 9110 S15.4.4 (303): See Other --- always GET on redirect
- RFC 9110 S15.4.8 (307): Temporary --- method preserved
- RFC 9110 S15.4.9 (308): Permanent --- method preserved
- Open redirect detection (haxgoat has =/open-redirect= but predicate does not
  detect unvalidated redirect targets)
- Redirect chain length (infinite redirect loops)
- Cross-origin redirect credential stripping
- Mixed-content redirect (HTTPS to HTTP downgrades mid-chain)
- =Location= header injection (CRLF injection in redirect target)

*Assessment*: Minimal. The HTTP-to-HTTPS downgrade check is useful but barely
scratches the surface. Open redirect detection is supported by haxgoat but
not tested by any predicate. Redirect semantics per status code (301 vs 302
vs 307 vs 308) are entirely absent.

** Cache Group

*** etag

*Normative specification*: RFC 9110 S8.8.3 (ETag), RFC 9110 S13 (Conditional
Requests)

*What hax checks*:
- ETag header presence only

*What the spec requires that hax does NOT check*:
- RFC 9110 S8.8.3: ETag must be either a strong validator (="etag"=) or weak
  (=W/"etag"=)
- RFC 9110 S8.8.3.2: Strong comparison vs weak comparison semantics
- RFC 9110 S8.8.1: ETag must change when the representation changes
- Interaction with =If-None-Match= (conditional GET)
- Interaction with =If-Match= (conditional PUT/DELETE for optimistic
  concurrency)
- ETag format validation (must be quoted string)
- Consistency: same resource, same representation, same ETag

*Assessment*: Presence-only. The real value of ETag testing is in conditional
request behavior (If-None-Match -> 304, If-Match -> 412), which requires the
sequential predicate type.

*** no-store

*Normative specification*: RFC 9111 S5.2.2.5 (no-store)

*What hax checks*:
- =Cache-Control= header contains =no-store=
- Warns if =Cache-Control= is absent

*What the spec requires that hax does NOT check*:
- RFC 9111 S5.2.1: Request directives vs response directives
- RFC 9111 S5.2.2.1: =must-revalidate= semantics
- RFC 9111 S5.2.2.2: =must-understand= semantics
- RFC 9111 S5.2.2.3: =no-cache= vs =no-store= distinction
- RFC 9111 S5.2.2.6: =private= vs =public= for shared caches
- RFC 9111 S5.2.2.4: =no-transform= (prevents proxies from modifying content)
- RFC 9111 S5.2.2.8: =proxy-revalidate= semantics
- RFC 9111 S5.2.2.10: =s-maxage= (shared cache max-age)
- RFC 9111 S5.3: =Expires= header and its interaction with =max-age=
- RFC 9111 S5.2.2.9: =max-age= value validation
- Sensitive data endpoints without =no-store= (context-dependent)
- =Pragma: no-cache= (HTTP/1.0 backward compatibility)

*Assessment*: Minimal. Checks for one directive out of approximately 12 defined
in RFC 9111. The =no-store= check is the correct starting point for
security-sensitive data, but Cache-Control analysis should be much broader.

*** vary

*Normative specification*: RFC 9110 S12.5.5 (Vary)

*What hax checks*:
- Vary header presence only

*What the spec requires that hax does NOT check*:
- RFC 9110 S12.5.5: Vary lists the request headers that affect content
  negotiation
- =Vary: *=  means every request is unique (disables caching)
- =Vary: Authorization= should be present on auth-dependent responses
- =Vary: Accept-Encoding= should be present when Content-Encoding varies
- =Vary: Origin= should be present when CORS headers vary by origin (critical
  for cache poisoning prevention)
- Cache poisoning via missing Vary headers (e.g., CORS response cached without
  =Vary: Origin= serves cross-origin response to same-origin request)

*Assessment*: Presence-only. The most security-critical Vary check is
=Vary: Origin= on CORS-enabled responses, which prevents CORS-based cache
poisoning. This is entirely absent.

*** 304

*Normative specification*: RFC 9110 S15.4.5 (304 Not Modified), RFC 9110 S13
(Conditional Requests)

*What hax checks*:
- Nothing. Returns "skip" unconditionally.

*What the spec requires*:
- Send GET, receive ETag/Last-Modified
- Send conditional GET with =If-None-Match= / =If-Modified-Since=
- Verify 304 response (no body, correct headers)
- Verify conditional PUT with =If-Match= returns 412 on stale ETag
- RFC 9110 S13.1.1: =If-Match= for lost-update prevention
- RFC 9110 S13.1.2: =If-None-Match= for cache validation
- RFC 9110 S13.1.3: =If-Modified-Since= for cache validation
- RFC 9110 S13.1.4: =If-Unmodified-Since= for conditional updates

*Assessment*: Completely unimplemented. Conditional requests are the foundation
of HTTP caching correctness and optimistic concurrency control. The haxgoat
server already supports 304 on =/cached-ok= but no predicate exercises it.

** State Group

*** workflow-skip

*Normative specification*: No RFC; application-level security. Relates to OWASP
A04 (Insecure Design) and A01 (Broken Access Control).

*What hax checks*:
- Nothing. Returns "skip" unconditionally.

*What the spec requires*:
- Attempt to access step N without completing steps 1..N-1
- Verify server rejects out-of-order requests
- haxgoat provides =/api/workflow/step1,2,3= for testing

*Assessment*: Unimplemented. This is Type 3 (sequential) and requires the
multi-request infrastructure. The design is sound but implementation is
missing.

*** toctou

*Normative specification*: No RFC; concurrency safety. Relates to CWE-367
(Time-of-check Time-of-use Race Condition).

*What hax checks*:
- Nothing. Returns "skip" unconditionally.

*What the spec requires*:
- Send N concurrent requests that exploit check-then-act patterns
- Verify that the server does not allow double-spend / double-action
- haxgoat provides =/api/race= with intentional TOCTOU vulnerability

*Assessment*: Unimplemented. Requires =repeat-concurrent= mutation operator
infrastructure. The executor already supports concurrent execution via
=ExecuteBatch= with =Concurrency > 1=.

*** replay

*Normative specification*: RFC 8470 (Using Early Data in HTTP, S5.1), OWASP
guidance on replay attacks, RFC 9110 S9.2.2 (idempotency as replay defense).

*What hax checks*:
- Nothing. Returns "skip" unconditionally.

*What the spec requires*:
- Send identical POST twice
- Verify second request is rejected or deduplicated
- Check for idempotency key headers (=Idempotency-Key=, =X-Request-Id=)
- RFC 8470 S5.1: 425 Too Early response for 0-RTT replay prevention

*Assessment*: Unimplemented. Replay detection is critical for financial and
state-changing APIs. The predicate should test both server-side deduplication
and the presence of idempotency key mechanisms.

* Prior Art: Exhaustive HTTP Compliance Testing

** h2spec (HTTP/2 Protocol Conformance)

- *URL*: https://github.com/summerwind/h2spec
- *Approach*: Exhaustive section-by-section test of HTTP/2 (RFC 7540/9113)
- *Structure*: Each test case maps to a specific RFC section number
- *Technique*: Sends malformed/edge-case frames, checks server response
- *Coverage*: 146 test cases covering nearly every MUST/SHOULD in the spec
- *Lessons for hax*:
  - h2spec proves that RFC-section-to-test-case mapping is viable
  - The naming convention (=3.5/1= for RFC section 3.5, test 1) makes
    traceability trivial
  - hax should adopt a similar naming: =rfc9110-9.2.1/1= for safety test 1
  - h2spec tests the protocol layer; hax tests the application behavior layer
  - Both share the "send adversarial input, check server conformance" pattern

** Web Platform Tests (WPT)

- *URL*: https://github.com/nicbarker/nicbarker/nicbarker/nicbarker/nicbarker/nicbarker/nicbarker/nicbarker/nicbarker
  (Correction: https://github.com/nicbarker/web-platform-tests ---
  canonical: https://wpt.live/)
- *Approach*: Browser conformance tests for web standards
- *Coverage*: ~55,000 tests across Fetch, CORS, CSP, Cookies, etc.
- *Relevance to hax*:
  - WPT tests browser behavior (client-side conformance)
  - hax tests server behavior (server-side conformance)
  - They are complementary: WPT verifies browsers enforce CORS;
    hax verifies servers configure CORS correctly
  - WPT's =fetch/= directory has extensive CORS tests that document
    edge cases hax should also cover
  - WPT's =content-security-policy/= tests enumerate CSP bypass techniques
  - hax could use WPT test descriptions as a predicate checklist

** Fetch API Tests (chromium)

- *URL*: Chromium source, =third_party/blink/web_tests/http/tests/fetch/=
- *Approach*: Integration tests for Chromium's Fetch implementation
- *Relevance*: Documents browser-side CORS/CSP enforcement behaviors
  that servers must be configured to trigger correctly

** Schemathesis

- *URL*: https://github.com/schemathesis/schemathesis
- *Approach*: Property-based testing of REST APIs using OpenAPI schemas
- *Technique*: Hypothesis-style generation of API requests from schema
- *Relevance to hax*:
  - Schemathesis operates at L2 (endpoint logic); hax at L1 (protocol axioms)
  - Schemathesis requires an OpenAPI spec; hax operates spec-free
  - Schemathesis generates inputs from schema constraints; hax generates inputs
    from protocol-level mutations
  - Key insight: Schemathesis stateful testing (link-based state machines) is
    analogous to hax's Type 3 predicates
  - Schemathesis proves PBT works for HTTP APIs at scale

** OWASP ZAP / Burp Suite (Scanner Approach)

- *URL*: https://www.zaproxy.org/, https://portswigger.net/burp
- *Approach*: Signature-based vulnerability scanning + active probing
- *Relevance to hax*:
  - ZAP's "passive scan" is analogous to hax's Type 1 predicates (header checks)
  - ZAP's "active scan" sends payloads --- analogous to hax's mutation operators
  - Key difference: ZAP uses vulnerability signatures (known-bad patterns);
    hax uses RFC-derived properties (known-good requirements)
  - ZAP finds specific CVEs; hax finds protocol non-compliance
  - hax's property-based approach is more fundamental but less
    immediately actionable

** Dredd (API Blueprint / OpenAPI Testing)

- *URL*: https://github.com/apiaryio/dredd
- *Approach*: Tests API responses against documentation
- *Relevance*: Contract testing for APIs; similar to hax's approach but
  schema-dependent

** RESTler (Microsoft Research)

- *URL*: https://github.com/microsoft/restler-fuzzer
- *Approach*: Stateful REST API fuzzer using grammar-based generation
- *Technique*: Infers producer-consumer dependencies between API calls
- *Relevance to hax*:
  - RESTler's stateful testing is directly relevant to hax's Type 3 predicates
  - RESTler discovers state-dependent bugs (workflow-skip, TOCTOU)
  - RESTler requires an OpenAPI spec; hax is spec-free
  - The producer-consumer dependency model could inform hax's workflow-skip
    predicate design

** Property-Based HTTP Testing: State of the Art

There is no existing tool that combines:
1. RFC-grounded properties (not signatures)
2. Structured mutation operators (not random fuzzing)
3. Spec-free operation (no OpenAPI required)
4. Property-based testing with shrinking

hax occupies a unique position. The closest approaches are:
- h2spec for protocol conformance (but HTTP/2 specific)
- Schemathesis for property-based API testing (but schema-required)
- RESTler for stateful testing (but schema-required)

The gap hax fills: RFC-grounded behavioral testing of HTTP endpoints without
requiring an API specification.

* Structural Recommendations

** 1. RFC Citation Discipline

Every predicate should include a doc comment citing the normative section:

#+begin_example
// checkFoo validates ... per:
//   - RFC NNNN SXXX: requirement description
//   - RFC NNNN SYYY: related requirement
#+end_example

Currently only =checkHSTS=, =checkIdempotencyMulti=, and =checkSafetyMulti=
follow this pattern. All 25 predicates should.

** 2. Predicate Naming Convention

Adopt h2spec-style names that encode the RFC section:

#+begin_example
rfc9110-9.2.1    (safety)
rfc9110-9.2.2    (idempotency)
rfc6797-6.1.1    (hsts-max-age)
rfc9111-5.2.2.5  (no-store)
#+end_example

This makes traceability automatic and gaps visible.

** 3. Test Matrix Completeness Check

Add a compile-time or test-time check that verifies:
- Every predicate in the code has a corresponding haxgoat endpoint
- Every haxgoat endpoint has at least one predicate that exercises it
- The relevance matrix references only existing predicates and groups

** 4. Presence vs Semantic Checks

The four presence-only predicates (corp, permissions-policy, etag, vary)
should all be upgraded to semantic checks. Presence-only is a scanner
pattern; hax should check behavior.

** 5. Stub Elimination Plan

The 6 stub predicates (retries, csrf, jsonp, 304, workflow-skip, toctou,
replay) have the infrastructure needed (MultiPredicate type, executor
with concurrency support) but no implementation. These should be
prioritized per the gap analysis above.
