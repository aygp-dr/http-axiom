#+TITLE: Five Persona Reviews: Synthesis
#+DATE: 2026-03-25

* Reviewers

1. *Skeptical CISO* — Fortune 500 security leader with Burp/ZAP/Snyk/Semgrep
2. *Go stdlib purist* — CLI craftsperson, Unix philosophy, 10K+ star projects

* Universal Agreement (all 5 concur)

- The *problem is real*: AI-generated code systematically omits HTTP compliance
- The *predicate model is sound*: RFC-grounded, three types, clean taxonomy
- The *7 stub predicates are misleading*: advertising checks that return "skip"
- *flag.FlagSet* should replace hand-written parsing (Go purist: "the stdlib is not a dependency")

* Most Damaging Critique: PBT Researcher

#+begin_quote
"The Go implementation is structured enumeration with deterministic
mutation, dressed in PBT vocabulary."
#+end_quote

Key points:
- Generator samples from 7 methods × 1 path × 1 auth = *7 points*. That's not PBT.
- Header predicates are *assertions* (~header present?~), not *properties* (~∀ input: invariant holds~)
- Shrink is correct but *not connected to execution pipeline*
- Stateful testing (spec.org RuleBasedStateMachine) has *zero Go implementation*
- *Verdict*: "10% of what the spec describes"

To become real PBT: use ~rapid~ (Go PBT library), generate from large spaces,
quantify over inputs, integrate shrink with execution.

* Most Actionable Critique: AI Coding Agent

#+begin_quote
"hax knows what is wrong. It needs to also say exactly how to make
it right, in the language of the framework being used."
#+end_quote

Three concrete product ideas:
1. *hax manifest* — export predicate knowledge as structured pre-generation context
2. *hax remediate* — return framework-specific fix snippets (Express/FastAPI/Rails/Go)
3. *hax serve --mcp* — MCP server for in-the-loop agent verification

The agent provided complete middleware snippets for 4 frameworks and then
admitted: "I am not 100% confident they pass every hax predicate." That's
the exact feedback loop hax should close.

* Most Practical Critique: DevOps Engineer

Pipeline adoption blockers (priority order):
1. No false-positive suppression (=.haxignore= / baseline)
2. Hardcoded auth credentials (executor uses ~Bearer test-token~)
3. No SARIF/JUnit output
4. No Docker image
5. Response body leak (V2) crashes runners at scale
6. Relevance matrix is dead code (V4)

*Math*: 200 services × 50 endpoints × 50ms latency = 67 min sequential.
Even parallelized: ~7 min, but V2 leak exhausts file descriptors first.

* Most Structural Critique: Go Purist

#+begin_quote
"When does 'no framework' become 'bad framework'? When you are
copy-pasting your own code within the same file."
#+end_quote

Key points:
- main.go at 1554 lines with *3 copies* of predicate dispatch logic
- ~sort.Strings~ avoided because "stdlib is a dependency" — this is religion, not engineering
- ~NamedPred~ with 3 optional function fields is a *sum type as product type* anti-pattern
- JSON array pipe contract prevents streaming — use NDJSON
- Global mutable state (~jsonOutput~, ~verboseMode~) prevents testing
- Error handling: 3 inconsistent patterns (fatal exit, skip+continue, silent drop)

* Most Strategic Critique: CISO

#+begin_quote
"Security headers are an infrastructure concern. In a Fortune 500,
HSTS is set at the load balancer, not in application code."
#+end_quote

hax needs to decide its user: developers testing locally (header failures
expected and noisy) or security teams validating infrastructure (duplicates
existing tools). The *interesting* predicates (CORS reflection, idempotency,
GET safety) go beyond header scanning — but those are the ones that are stubs.

* Identity Clarification

The reviews converge on an identity question. hax is NOT:
- A security scanner (that's ZAP/Burp)
- A header checker (that's SecurityHeaders.com)
- A compliance tool (that's SARIF-speaking enterprise scanners)

hax IS:
- A *specification testing tool* for HTTP protocol properties
- A *property-based testing framework* for RFC invariants (once ~rapid~ is integrated)
- A *verification gate* for AI-generated code
- A *pre-generation knowledge source* for AI agents

The rebuild should lead with "RFC compliance via property-based testing"
not "security scanning."

* Priority Tiers for Rebuild

** P0: Core identity (must have for v2 to be v2)

- ~rapid~ integration for real PBT generation + shrinking
- ~flag.FlagSet~ per command (stdlib, prevents all 5 CLI bugs)
- Remove stub predicates from public interface (don't advertise "skip")

** P1: Primary user value (AI agents are the user)

- ~hax manifest~ — export predicate knowledge as structured pre-generation context
- ~hax remediate~ — framework-specific fix snippets per finding
- Wire relevance matrix into cmdRun (stop testing 280 nonsense combinations)

** P2: Correctness (contract violations from audit)

- Fix V2: close all ~Responses[]~ not just ~[0]~
- Fix V3: close doctor HEAD response
- Fix V4: relevance matrix dead code (overlaps with P1)
- Explicit predicate Type enforcement (no Fn+MultiFn ambiguity)

** P3: Pipeline adoption (nice to have, not core)

Everything below is polish. It matters for enterprise adoption but
not for the tool's identity or correctness. Defer until the core
(P0-P2) is solid.

- SARIF output format (~--format sarif~)
- JUnit XML output (~--format junit~)
- Config file (~.haxrc~ / ~hax.yaml~) with per-predicate suppression
- False-positive baselining (~hax baseline~ command)
- Real auth credential injection (~--header~, ~--bearer-token~, env vars)
- Docker image (multi-stage, scratch-based)
- GitHub Actions workflow template
- NDJSON pipe contract for streaming
- Rate limiting / throttling controls
- Scope control (~--include~, ~--exclude~ paths)
- ~internal/output~ presentation tier refinement
- MCP server mode (~hax serve --mcp~)
- Stateful PBT via ~rapid.MakeMachine~
- Coverage-guided generation
- Defect Dojo / SIEM integration formats

These are all valid requirements from the CISO and DevOps reviews.
They are not rejected — they are *deferred* because they are adoption
features, not identity features. A tool that does real PBT over RFC
specs with remediation output for AI agents does not need SARIF to
justify its existence. SARIF helps it *scale*. The core must work first.
